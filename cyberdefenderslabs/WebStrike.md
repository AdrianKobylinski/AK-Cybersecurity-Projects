# WebStrike - CyberDefenders Write-up

## Tools Used

* Wireshark
* GeoIP Lookup

## Scenario

A suspicious file was discovered on a company web server, raising concerns about potential malicious activity within the organization's network. The Development Team reported the anomaly, and the Network Team captured network traffic and provided a PCAP file for investigation.

The objective of this analysis was to determine how the file appeared on the server, identify the attacker, and assess the extent of the compromise.

---

# Question 1

## From which city did the attack originate?

My first step was to identify the hosts involved in the communication. By reviewing the network conversations in Wireshark, I determined that the PCAP contained traffic between only two IP addresses.

To identify the attacker, I analyzed the HTTP traffic and focused on the requests generated by each host. After filtering for HTTP GET requests, I observed that one IP address was probing several sensitive directories, including:

* `/admin`
* `/uploads`
* `/reviews/uploads`

This behavior is commonly associated with directory enumeration and reconnaissance activity, indicating an attempt to discover accessible resources and potential attack vectors.

Based on this activity, I identified **117.11.88.124** as the attacking host and **24.49.63.79** as the victim web server.

Following the challenge requirements, I performed a GeoIP lookup on the attacker's IP address to determine its geographic location.

# Question 3

## What is the name of the malicious web shell that was successfully uploaded?

To determine whether the attacker exploited a vulnerability, I investigated the HTTP POST requests associated with file uploads.

During the analysis, I identified a suspicious file that had been uploaded to the server. The file used a double extension, a common technique used to bypass file upload restrictions while still allowing the file to be executed as PHP code.

The attacker later accessed the uploaded file directly, confirming that the upload was successful and that the file was being used as a web shell.

# Question 5

## Which port, opened on the attacker's machine, was targeted by the malicious web shell?

To identify the communication channel used after exploitation, I examined the commands executed through the web shell.

The traffic revealed a reverse shell payload utilizing a named pipe (`/tmp/f`) and `/bin/sh`, which attempted to establish a connection back to the attacker's machine.

By analyzing the command, I identified the destination port used for the unauthorized outbound connection.

### Answer

# Question 6

## Which file was the attacker attempting to exfiltrate?

To identify the targeted data, I reviewed the HTTP POST requests generated after the successful compromise.

Using the following Wireshark filter:

```text
http.request.method == "POST"
```

I observed activity indicating that the attacker attempted to access and transfer a sensitive Linux system file.

# Conclusion

The investigation revealed a complete attack chain beginning with reconnaissance and directory enumeration against the target web application.

The attacker successfully identified a vulnerable file upload functionality and uploaded a malicious PHP web shell named **image.jpg.php**. The web shell was stored within the **/reviews/uploads/** directory and later executed to gain remote command execution on the compromised server.

Further analysis showed that the attacker attempted to establish a reverse shell connection to their machine on port **8080**, providing persistent remote access. Finally, the attacker attempted to access and exfiltrate the **/etc/passwd** file, which contains information about user accounts on Linux systems.

This investigation demonstrates a typical web application compromise involving reconnaissance, exploitation of insecure file uploads, remote command execution, and attempted data exfiltration.
